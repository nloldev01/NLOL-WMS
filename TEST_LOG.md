## Bugs found

### BUG-001 — Several ViewSets across 3 apps have NO permission checks at all (Security, Critical)
- **Confirmed by walking the entire URL resolver and checking `permission_classes` on every registered view** (not just spot-checks) — full exposure list:
  - **`master_data`** — every ViewSet: `UnitViewSet`, `FiscalYearViewSet`, `AssetViewSet`, `LocationViewSet`, `RawMaterialAndConsumableViewSet`, `ProductGroupsViewSet`, `ProductSubGroupViewSet`, `ProductSegmentViewSet`, `ProductViewSet`, `FinishedProductViewSet`, `FinishedProductVariantViewSet`, `SupplierViewSet`.
  - **`packaging`** — `PackagingOrderViewSet` (including its `start`, `complete`, `cancel` actions).
  - **`production`** — `ProductionOrderViewSet` and **every one of its actions**: `update-materials`, `update-actual-qty`, `start-mixing`, `toggle-material-loaded`, `confirm-materials`, `complete-order`. Notably `RecipeViewSet` in the *same file* correctly uses `ModulePermission.require('production_recipes')` — only the order viewset was missed.
- **Root cause**: `REST_FRAMEWORK` in `nlol_wms/settings.py` has no `DEFAULT_PERMISSION_CLASSES`, so DRF's global default (`AllowAny`) silently applies to any ViewSet that doesn't explicitly set `permission_classes` — there's no safety net.
- **Repro**: `GET http://localhost:8000/api/master-data/units/` or `GET .../production/orders/` with no `Authorization` header → `200 OK` with full data (verified live for both).
- **Impact**: Anyone, with no login at all, can read **and write** (POST/PUT/PATCH/DELETE) all master data, packaging orders, and production orders — including starting/completing production runs, which also drives raw-material consumption.
- **Status**: Logged, not fixed yet (user chose to keep testing first, fix later).
- **Fix direction**: Two layers — (1) add `permission_classes = ModulePermission.require(...)` to each listed ViewSet to match the existing pattern used everywhere else, and (2) set `DEFAULT_PERMISSION_CLASSES = ['rest_framework.permissions.IsAuthenticated']` in settings as a safety net so a future missed ViewSet fails closed instead of open.
- **Note**: The bare `api/<app>/` "API root" listing pages also show as open — that's normal DRF `DefaultRouter` behavior (just lists endpoint names, no data) and isn't a real issue.
- **FIXED** (2026-06-19): Added `DEFAULT_PERMISSION_CLASSES = ['rest_framework.permissions.IsAuthenticated']` to `REST_FRAMEWORK` in `nlol_wms/settings.py` as a fail-closed safety net. Explicitly marked the 5 endpoints that must stay public as `AllowAny` (`login_view`, `setup_2fa_view`, `verify_2fa_setup_view`, `verify_2fa_login_view` in `accounts/views.py`; `logout_view` in `users/views.py`). Added `permission_classes = ModulePermission.read_write('master_data')` to all 12 `master_data` ViewSets, `ModulePermission.read_write('packaging')` to `PackagingOrderViewSet`, and `ModulePermission.read_write('production')` to `ProductionOrderViewSet`. Verified live: previously-open endpoints now return 401 with no token and 200 with a valid token; `python manage.py test` still passes (6/6). Full resolver walk confirms zero remaining unintentionally-open routes.
- **Follow-up note**: the `user` role currently has `none` access to `master_data`/`packaging`/`production` in `RoleModulePermission` (pre-existing data, unrelated to this fix) — worth checking with the business whether regular `user`-role accounts need at least `view` access to master data once this is deployed, since that permission is now actually enforced for the first time.

### BUG-002 — 2FA enrollment is unreachable / locks the user out (Logic, High)
- **Where**: `users/views.py:toggle_2fa` (line ~96-112) and `accounts/views.py:setup_2fa_view` (line ~92-93).
- **Repro**:
  1. Superadmin calls `POST /api/users/<id>/toggle-2fa/` → sets `is_2fa_enabled=True` immediately, no OTP secret yet.
  2. That user logs in (`POST /api/login/`) → since `is_2fa_enabled` is now `True`, backend correctly returns a `temp_token` for setup.
  3. Frontend/client calls `POST /api/setup-2fa/` with that `temp_token` → **fails with `{"error":"2FA is already enabled"}`** because `setup_2fa_view` rejects whenever `user.is_2fa_enabled` is already `True`.
- **Impact**: There is no code path where `setup_2fa_view` can ever succeed. `toggle_2fa` sets the flag *before* a secret exists, but `setup_2fa_view` requires the flag to still be `False` to proceed. Any user an admin enables 2FA for is permanently locked into a "2FA required but un-completable" state — they can log in just far enough to get a `temp_token`, but can never retrieve a QR code/secret or verify a code, so `verify-2fa-login` will also always fail (no `otp_base32_secret` was ever generated).
- **Fix direction**: `setup_2fa_view` should allow setup when `is_2fa_enabled=True` AND `otp_base32_secret` is not yet confirmed-via-verification — i.e. the "already enabled" guard needs a separate signal (e.g. only block if verification already happened) rather than reusing the same flag `toggle_2fa` sets up front. Equivalently, `toggle_2fa` could use a distinct "pending setup" state instead of flipping `is_2fa_enabled` straight to `True`.
- **Status**: Logged, not fixed yet.

### BUG-003 — Creating a new Raw Material/Consumable crashes with 500 IntegrityError (Functional, High)
- **Where**: `POST /api/master-data/raw-materials-and-consumables/`
- **Repro**: `POST` with valid fields per the current model/serializer (`{"name":"QA Raw Material","type":"raw_material","unit":11}`) → **500 Internal Server Error**, not a clean 400.
- **Root cause**: `IntegrityError: null value in column "category" of relation "master_raw_materials_and_consumables" violates not-null constraint`. The Postgres table has a NOT NULL `category` column that does **not exist** in the current `RawMaterialAndConsumable` Django model (`master_data/models.py`) or its serializer — schema drift. No migration in `master_data/migrations/` ever adds a `category` column to this table, meaning it was added directly to the DB outside Django's migration system (or a field was removed from the model without a matching `RemoveField`/default-backfill migration).
- **Impact**: No new raw material or consumable can be created through the API at all — every attempt 500s. This blocks onboarding any new raw material, which the rest of the Raw Materials Stock and Production walkthrough depends on (existing seeded rows already have `category` populated from before this drift, which is why it wasn't caught earlier).
- **Fix direction**: Either drop the stray `category` column (if truly unused) via a proper migration, or add `category` back to the model + serializer with a sensible default/migration so existing rows and new rows stay consistent.
- **Status**: Logged, not fixed yet. Continuing remaining tests against existing seeded materials (e.g. id=2 "Engine Oil") to avoid being blocked.

### BUG-004 — Misleading "Insufficient stock" error when `batch` is omitted on usage (UX, Medium)
- **Where**: `POST /api/raw-materials-stock/stock-movements/record/` (`movement_type=usage`/etc.), backing logic in `RawMaterialStockLog.create_movement`.
- **Repro**: Material has 100 units in stock under `batch=133` at a location. Recording a `usage` movement for the *same material+location* **without** passing `batch` returns `400 {"error":"Insufficient stock: 0 available, 30 requested..."}` — even though 100 units genuinely exist (just under a specific batch). Passing `batch=133` explicitly succeeds and decrements correctly (100 → 70).
- **Why it's not severity-High**: the real `StockMovementPage.jsx` UI always presents a batch dropdown for outbound movements, so a user going through the normal UI flow would select a batch. Risk is limited to: (a) the error message itself is misleading (says "stock shortage" when the real issue is "no batch specified, so it looked at an empty/no-batch row"), and (b) any future API consumer or admin/API tool that doesn't realize batch is effectively required would hit a confusing false negative.
- **Fix direction**: either default to FIFO-consuming the oldest batch when `batch` isn't supplied, or return a clearer error like "Please specify a batch" when stock exists under other batches but none was specified for an outbound movement at that location.
- **Status**: Logged, not fixed yet.

### BUG-005 — Negative `target_quantity` accepted on Production Order creation (Validation, Medium)
- **Where**: `POST /api/production/orders/`, `ProductionOrderSerializer`/`ProductionOrder` model (`production/models.py`, `production/serializers.py`).
- **Repro**: `POST` with `{"recipe":2,"kettle":18,"target_quantity":-5}` → `201 Created`, order saved with `target_quantity: "-5.0000"` and its auto-populated material line computed as `planned_qty: "-2.5000"` (negative planned consumption).
- **Impact**: Nonsensical data enters the system uncaught; downstream `complete-order` logic does decimal arithmetic on these quantities (consumption, wastage, stock balance updates) which is unverified for negative inputs and could corrupt stock balances in unpredictable ways if such an order were ever started/completed.
- **Fix direction**: Add a `MinValueValidator(0)` (or stricter, `>0`) on `ProductionOrder.target_quantity`, enforced at the serializer level so the API rejects it with a clean 400 rather than the model layer silently accepting it.
- **Status**: Logged, not fixed yet. Test order (id=37) was deleted after confirming the bug, no lasting data impact.

### BUG-006 — `Invoice.net_amount` is never computed server-side, silently defaults to 0 (Data integrity, High)
- **Where**: `sales/models.py` `Invoice.net_amount` (plain `DecimalField(default=0)`, no `save()` override), `sales/serializers.py` `InvoiceSerializer`/`InvoiceDetailSerializer` (fully writable, not computed), `sales/views.py` bulk CSV/Excel upload paths (lines ~181, ~550 — read `net_amount` straight from the uploaded file's column).
- **Repro**: `POST /api/sales/invoices/` with `{"gross_amount":1000,"discount":100,...}` (no `net_amount`) → `201` with `net_amount: "0.00"`, not the expected `900.00`.
- **Why it's worse than a one-off API quirk**: `frontend/src/pages/SalesPage.jsx` never references `net_amount` at all, so the manual invoice-creation UI never sends it either — every manually-created invoice gets `net_amount=0` silently. The bulk-upload path doesn't compute it from gross/discount either; it just trusts whatever "Net Amount" column value is in the uploaded file, with no cross-check.
- **Impact**: `net_amount` feeds revenue reporting directly (`dashboard_stats` aggregates `Sum('net_amount')`, `Avg('net_amount')` in `sales/views.py` ~619-636). A manually-entered invoice silently reports $0 revenue while still showing the correct gross/discount, and there's no validation anywhere that `net_amount == gross_amount - discount` for uploaded data either — a typo'd spreadsheet column would silently corrupt reported revenue with no error.
- **Fix direction**: Compute `net_amount = gross_amount - discount` server-side (e.g. in `Invoice.save()` or the serializer's `create`/`update`), or at minimum validate that a client-supplied `net_amount` matches that formula and reject mismatches.
- **Status**: Logged, not fixed yet. Test invoice/customer created during this check were deleted, no lasting data impact.
